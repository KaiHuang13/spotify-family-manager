# 系統架構

## 1. 文件目的

本文件描述第一階段的技術方向與系統邊界，作為後續畫面設計、資料庫設計及實作的依據。目前不包含可執行程式碼或實際資料庫建置。

## 2. 技術選型

| 層級   | 規劃技術                        | 用途               |
| ---- | --------------------------- | ---------------- |
| 前端   | React + TypeScript          | 管理介面與型別安全        |
| 建置工具 | Vite                        | 開發與靜態檔案建置        |
| 靜態部署 | GitHub Pages                | 發布前端網站           |
| 身分驗證 | Supabase Auth               | 管理者登入與工作階段       |
| 資料庫  | Supabase PostgreSQL         | 保存訂閱、成員、帳期及付款資料  |
| 資料授權 | Supabase Row Level Security | 由資料庫層限制管理者可存取的資料 |

## 3. 系統邊界與資料流

1. 管理者從部署於 GitHub Pages 的前端網站登入。
2. 前端透過 Supabase Auth 取得已登入使用者身分與工作階段。
3. 前端以使用者權杖直接呼叫 Supabase 資料服務。
4. PostgreSQL 的 RLS 政策依登入使用者識別碼判斷資料讀寫權限。
5. 管理者在系統外收款，再於前端手動登記付款。
6. 儀表板與統計資料由受 RLS 保護的帳期及付款資料產生。

系統不處理金流、不連接銀行，也不需要存取 Spotify API。

## 4. 建議前端模組

- `auth`：登入、登出、工作階段與未授權導向。
- `dashboard`：訂閱摘要、名額、應收與逾期概況。
- `members`：成員基本資料、付款週期與加入／離開管理。
- `subscriptions`：服務與方案設定、價格及生效期間。
- `billing`：帳期、應收金額、到期與狀態。
- `payments`：手動付款登記、更正及付款歷史。
- `statistics`：期間彙總與趨勢。
- `history`：關鍵異動與帳務歷史。
- `shared`：共用型別、格式化、驗證與 UI 元件。

這些是責任邊界，不代表目前要建立對應資料夾或程式碼。

## 5. 概念資料模型

實際欄位、關聯與約束需在業務規則確認後設計。第一階段可考慮以下實體：

### `profiles`

對應 Supabase Auth 使用者，保存系統需要的管理者資料。資料以 Auth 使用者識別碼為擁有者依據。

### `services`

描述訂閱服務。第一階段只有 Spotify，但保留未來加入其他服務的能力。

### `subscriptions`

描述管理者持有的方案、總名額、幣別、費用、扣款週期、狀態及有效期間。

### `members`

保存成員基本資料、狀態、加入日與離開日。應包含資料擁有者識別，以便 RLS 隔離。

### `member_plans`

描述成員在特定期間的月繳／年繳選擇、分攤方式、金額及生效期間。獨立保存可避免設定變更覆寫歷史。

### `billing_periods`

描述一個方案的帳務期間，可作為同期間成員應收的彙整依據。

### `charges`

保存個別成員在特定帳期的應收金額、到期日與狀態。應將當期計算結果固化，以保護歷史資料。

### `payments`

保存管理者手動登記的付款日期、金額、方式、備註及其對應應收。

### `audit_events`

保存需追溯的關鍵事件，例如成員加入／離開、費率變更、付款更正及帳期作廢。是否需要完整欄位級稽核待確認。

## 6. 資料模型原則

- 業務資料應有 `owner_id` 或可安全追溯至擁有者的關聯。
- 金額使用 PostgreSQL 適當的精確數值型別或最小貨幣單位，不使用浮點數。
- 日期與時間需區分「帳務日期」與「事件時間」；事件時間以含時區格式保存。
- 方案價格及成員分攤設定採生效期間管理，不回溯覆寫歷史應收。
- 關聯完整性、唯一性與非負金額等重要規則應由資料庫約束保護。
- 帳期狀態若可由應付、實收及到期日推導，需決定採即時計算或受控同步，避免多份狀態不一致。
- 財務性資料優先作廢或更正，不直接硬刪除。

## 7. 身分驗證與 RLS 原則

- 所有業務資料表預設拒絕存取並啟用 RLS。
- 已登入使用者只能存取屬於自身 `auth.uid()` 的資料。
- 新增資料時，資料擁有者必須等於目前登入使用者。
- 更新與刪除操作需同時檢查既有列及更新後資料的擁有者。
- 不依賴前端傳入的任意管理者 ID 作為授權依據。
- 前端只使用 Supabase 公開金鑰；service role key 不得進入前端、Git Repository 或 GitHub Pages。
- 第一階段雖只有一位管理者，資料仍以擁有者隔離模型設計，避免未來擴充時重做安全邊界。
- RLS 政策需有正向與反向測試：本人可操作、未登入者不可操作、其他使用者不可操作。

## 8. GitHub Pages 部署考量

- Vite 的 base path 需符合 Repository Pages 的子路徑。
- 前端路由可採 hash routing，或另行設計適用 GitHub Pages 的 fallback；方案待實作前確認。
- Supabase 登入後的 redirect URL 必須加入正式 Pages 網址及本機開發網址。
- 部署環境變數只能包含可公開的前端設定。
- GitHub Pages 僅提供靜態前端；所有受保護資料操作由 Supabase Auth、資料庫約束與 RLS 負責。

## 9. 統計與歷史策略

- 第一階段優先由 `charges` 與 `payments` 聚合產生統計，確保與明細一致。
- 若資料量增加後需要效能優化，再評估資料庫 view、materialized view 或受控彙總表。
- 統計須明確區分應收、實收、未收與逾期，並確認期間歸屬採帳期或付款日。
- 歷史事件與帳務交易分開：帳務表保存事實，稽核事件說明何時發生何種變更。

## 10. 後續架構決策

開始實作前仍需決定：

- 登入方式採 Email 密碼、Magic Link 或其他 Supabase Auth 方法。
- GitHub Pages 的正式路徑與前端路由策略。
- 帳期產生方式：前端按需產生、資料庫函式產生，或排程產生。
- 部分付款及一筆付款跨多筆應收的資料關聯方式。
- 歷史紀錄的詳細程度與保存期限。
- 是否需要附件儲存；若需要，需納入 Supabase Storage 與對應存取政策。
- 統計期間採現金基礎、應計基礎或同時支援。
