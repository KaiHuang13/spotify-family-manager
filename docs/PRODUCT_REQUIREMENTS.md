# Spotify Premium Family 訂閱與費用分攤管理系統—產品需求文件

## 1. 文件目的

本文件定義第一階段產品目標、使用者、功能範圍與驗收方向，作為設計、開發與測試的共同基準。

## 2. 產品背景與問題

Spotify Premium Family 的管理者需要掌握成員加入與離開時間、各期應分攤費用、實際付款狀態及歷史紀錄。以試算表或聊天訊息管理容易出現帳期不一致、漏記付款、歷史資料被覆蓋與統計困難等問題。

本系統提供單一管理介面，協助管理者以可追溯的方式維護訂閱與收付款資訊。第一階段不代收款、不串接 Spotify API，也不取代正式會計系統。

## 3. 產品目標

- 讓管理者快速看見目前訂閱、成員與付款狀態。
- 依成員的訂閱期間與繳費週期建立可追溯的費用紀錄。
- 保留成員、帳期、應收費用與付款的歷史，避免停用成員造成資料遺失。
- 支援月繳與年繳情境，並清楚呈現未付、部分付款、已付及逾期狀態。
- 建立未來擴充其他訂閱服務的資料與產品基礎，但不在第一階段實作多服務管理。

## 4. 使用者與權限

第一階段只有一種使用者角色：管理者。

- 系統初期僅供一位管理者使用。
- 管理者需透過 Supabase Auth 登入後才能存取資料。
- 所有業務資料均屬於該管理者；資料存取仍須以 PostgreSQL RLS 保護，不能因只有單一使用者而省略授權設計。

## 5. 第一階段範圍

### 5.1 訂閱狀態儀表板

儀表板至少呈現：

- Spotify Premium Family 訂閱目前狀態與當期帳期。
- 啟用中及已停用的成員數。
- 當期應收總額、已收總額、未收總額。
- 未付款、部分付款、逾期的成員或費用筆數。
- 接近到期的年繳成員提醒。

### 5.2 成員管理

- 新增成員基本資料。
- 修改成員顯示名稱、聯絡資訊與備註。
- 將成員停用，並記錄停用日期。
- 顯示成員目前狀態及完整歷史。
- 已存在訂閱、費用或付款歷史的成員不得直接刪除。

### 5.3 成員訂閱期間

- 記錄成員加入日期與結束日期。
- 同一成員可保留多段訂閱期間，以支援退出後重新加入。
- 訂閱期間不可出現結束日早於開始日。
- 同一成員的有效訂閱期間原則上不得重疊；例外需明確記錄原因。

### 5.4 月繳與年繳

- 每段成員訂閱可選擇月繳或年繳。
- 記錄週期價格、貨幣與生效期間。
- 變更價格或繳費週期時保留舊紀錄，不覆寫已產生的歷史費用。
- 年繳到期與續期需能獨立追蹤。

### 5.5 帳期與費用分攤

- 建立 Spotify Family 的帳期，包含起訖日、到期日與該期總成本。
- 將帳期費用分配至符合條件的成員，形成個別應收費用。
- 每筆分攤結果保留當時的金額與計算依據快照；日後調價不回溯改寫。
- 第一階段採管理者可確認與調整的方式，不要求全自動產生帳單。
- 部分月份加入、退出、四捨五入與剩餘差額規則在實作前須記錄於 `docs/DECISIONS.md`。

### 5.6 付款紀錄與付款狀態

- 管理者可手動新增付款，包含付款人、金額、付款日期、方式、備註與參考資訊。
- 一筆付款可分配至一筆或多筆應收費用；一筆應收費用也可接受多次付款。
- 系統依應收金額、已分配付款及到期日顯示未付款、部分付款、已付款或逾期。
- 更正付款時需保留可追溯性；不可因修改而讓歷史總額無法核對。
- 第一階段不串接任何金流、銀行或電子支付服務。

### 5.7 統計與歷史紀錄

- 依月份或自訂期間查看應收、已收、未收及訂閱成本。
- 查看各成員的訂閱期間、應收費用與付款歷史。
- 查看帳期歷史及各期分攤結果。
- 關鍵資料異動應留下稽核資訊，例如建立時間、修改時間與操作者；完整稽核事件的實作深度依開發計畫逐步完成。

## 6. 核心業務規則

- 所有金額以最小貨幣單位整數儲存與運算，禁止使用浮點數。
- 貨幣代碼需明確儲存；第一階段可預設單一貨幣，但資料不得只靠畫面假設。
- 付款紀錄不等同應收費用；兩者須透過付款分配關聯，以支援部分付款與合併付款。
- 已產生的應收費用必須保留金額快照，不隨成員目前方案或價格變更。
- 停用成員不再參與新帳期，但既有歷史仍可查詢與統計。
- 付款狀態應由應收、付款分配與到期日推導，避免同時維護互相矛盾的手動狀態。
- 日期區間的起訖是否包含端點，必須在實作前統一；建議採「開始日包含、結束日不包含」的半開區間。

## 7. 非功能需求

### 安全性

- 使用 Supabase Auth 驗證管理者身分。
- 所有業務資料表啟用 RLS，並驗證只能存取擁有者資料。
- 前端不得包含 Supabase `service_role` key 或其他高權限秘密。
- 敏感設定透過部署環境變數管理，且不得提交至版本控制。

### 正確性與可追溯性

- 金額運算必須可重現，且能核對帳期總額、成員應收與付款分配。
- 重要歷史資料採停用、沖銷或更正紀錄，不以硬刪除破壞關聯。
- 資料庫結構變更一律透過 Migration。

### 可用性

- 主要管理流程應支援桌面與行動裝置瀏覽。
- 金額、日期、付款狀態與錯誤訊息須清楚，不以顏色作為唯一辨識方式。
- 常見操作需有確認與輸入驗證，避免誤停用或重複登記付款。

### 部署與相容性

- 前端使用 React 與 TypeScript，部署至 GitHub Pages。
- 路由與資源路徑需相容 GitHub Pages 的 Repository base path 與重新整理行為。
- Supabase Auth redirect URL 需涵蓋正式 GitHub Pages 網址與核准的本機開發網址。

## 8. 不在第一階段範圍

- 建立或管理多位系統管理者與複雜角色權限。
- 串接 Spotify API 或自動同步 Spotify Family 成員。
- 串接金流、銀行、信用卡、電子支付或自動對帳。
- 自動寄送催款通知、Email、簡訊或通訊軟體訊息。
- 管理 Spotify 以外的訂閱服務。
- 提供成員自行登入與查看帳務的入口。
- 正式會計、發票、稅務或報稅功能。

## 9. 第一階段完成判準

- 管理者可安全登入，未登入者無法讀寫業務資料。
- 管理者可新增、修改、停用成員，且歷史資料不因停用而消失。
- 可建立月繳或年繳的成員訂閱期間。
- 可建立帳期與成員應收費用，且金額使用整數處理。
- 可手動登記與分配付款，並正確呈現付款狀態。
- 儀表板、統計與歷史頁面能從同一組資料得到一致結果。
- RLS、主要業務規則與核心計算均有測試。
- 專案通過 lint、typecheck、test，並可部署至 GitHub Pages。

## 10. 待確認事項

- Spotify Family 每期實際總成本、貨幣與調價處理方式。
- 費用採平均分攤、固定成員價格，或允許兩者並存。
- 成員在帳期中途加入或退出時是否按日比例計算。
- 除不盡的最小貨幣單位差額如何分配。
- 年繳是預收未來期間、支付固定十二個月，或僅作為付款提醒週期。
- 付款更正採編輯、作廢重建或沖銷紀錄的正式規則。
