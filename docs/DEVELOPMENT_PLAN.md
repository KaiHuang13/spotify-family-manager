# 開發計畫

## 1. 目的與原則

本計畫將第一階段切分為可獨立驗證的工作項目。每次只執行一個明確任務；開始前檢查既有架構，完成後執行 lint、typecheck、test，並回報修改檔案、驗證結果與未處理事項。

本文件是計畫，不代表已授權立即建立應用程式、安裝套件或部署。本次文件建立完成後，後續開發應再逐項執行。

## 2. 建議交付順序

### 階段 0：需求與規則定案

目標：消除會直接影響資料模型與計算的歧義。

- 確認貨幣、Spotify 方案成本與計費期間邊界。
- 確認平均分攤、固定價格及中途加入／退出的計算規則。
- 確認月繳、年繳的業務語意與到期提醒方式。
- 確認付款更正與差額分配規則。
- 將決策新增至 `docs/DECISIONS.md`，必要時同步更新產品需求與資料模型。

完成條件：所有會影響 Migration 或費用計算的待確認事項皆已決定，或明確延後且不阻礙下一階段。

### 階段 1：專案骨架與品質基線

目標：建立最小可執行的 React + TypeScript 專案與一致的驗證流程。

- 建立 React、TypeScript 的前端骨架與開發／建置設定。
- 設定 lint、typecheck、test 指令及基本測試環境。
- 設定環境變數範例，不放入任何秘密。
- 設定適用 GitHub Pages 的 base path 與路由策略。
- 建立最小 CI，執行 lint、typecheck、test 與 build。

完成條件：本機與 CI 可執行全部品質檢查，空白應用程式可正確建置。

### 階段 2：Supabase 基礎與安全性

目標：建立可重現且預設安全的資料層。

- 建立 Supabase 專案設定與本機開發方式。
- 以 Migration 建立第一版資料表、限制、索引與時間戳記。
- 建立管理者 profile 與資料擁有者欄位。
- 為所有業務資料表啟用 RLS 並建立最小權限 Policy。
- 建立 Auth 登入、登出與 session 恢復流程。
- 測試未登入拒絕存取、已登入僅能存取自己的資料。

完成條件：Migration 可從空資料庫重建結構，RLS 測試通過，前端只使用公開用戶端金鑰。

### 階段 3：成員與訂閱期間

目標：完成成員生命週期及月繳／年繳設定。

- 成員清單、新增、修改與停用。
- 成員詳情與多段訂閱期間。
- 月繳／年繳、價格、貨幣、生效與結束日期。
- 防止無效或不允許的訂閱期間重疊。
- 確保已有歷史資料的成員無法被直接刪除。

完成條件：核心流程與驗證測試通過，停用成員的歷史仍完整可查。

### 階段 4：計費期間與費用分攤

目標：建立可重現、可核對的應收費用。

- 建立與管理月份費用（資料語意為計費期間）。
- 依已確認規則選出參與分攤的成員。
- 計算個別應收費用，保存計算與價格快照。
- 顯示分攤合計與月份費用總額的差異。
- 提供管理者確認及必要的人工調整，保留調整理由。
- 避免同一計費期間重複產生相同費用。

完成條件：整數金額計算、邊界日期、四捨五入／差額及重複執行等測試通過。

### 階段 5：付款與狀態

目標：支援管理者手動登記及核對付款。

- 新增付款紀錄。
- 將付款分配至一筆或多筆應收費用。
- 支援部分付款、合併付款與超額輸入防護。
- 由金額與到期日推導未付、部分付款、已付及逾期狀態。
- 實作已確認的付款更正或作廢方式。

完成條件：付款總額、分配總額與各應收餘額可相互核對，狀態推導測試通過。

### 階段 6：儀表板、統計與歷史

目標：提供一致、可操作的管理資訊。

- 訂閱狀態、成員數與本期摘要。
- 未付、部分付款、逾期與年繳到期提醒。
- 期間應收、已收、未收與成本統計。
- 成員、月份費用、付款與調整歷史。
- 針對資料量增加後的查詢建立必要索引或資料庫 View。

完成條件：摘要數值可追溯至明細，空資料與跨期情境顯示正確。

### 階段 7：可用性、安全與部署驗收

目標：完成第一階段上線準備。

- 響應式版面、鍵盤操作、表單錯誤與非純色狀態提示。
- Auth redirect、GitHub Pages 路由與 base path 驗證。
- 檢查建置產物與 Repository 不含秘密或 `service_role` key。
- 執行完整 lint、typecheck、test、build 與關鍵流程驗收。
- 建立部署與回復說明，部署至 GitHub Pages 測試環境或正式環境。

完成條件：符合 `docs/PRODUCT_REQUIREMENTS.md` 的第一階段完成判準，部署後可完成主要管理流程。

## 3. 測試策略

- 單元測試：金額計算、計費期間邊界、付款狀態、日期與格式化工具。
- 整合測試：Migration、限制條件、RLS、月份費用產生應收、付款分配。
- 元件測試：重要表單驗證、狀態顯示、確認操作。
- 端對端測試：登入、建立成員、設定訂閱、產生費用、登記付款、查看儀表板。
- 部署驗證：GitHub Pages 深層連結或 hash route、重新整理、靜態資源與 Auth redirect。

每個階段都需補足該階段的測試，不把所有測試延後至最後。

## 4. 任務完成範本

每個開發任務完成時，回報：

1. 任務目標與完成結果。
2. 修改檔案。
3. Migration 及資料相容性影響（若有）。
4. lint、typecheck、test 的實際結果。
5. 手動驗證結果（若適用）。
6. 尚未處理事項與已知風險。

## 5. 主要風險與因應

| 風險 | 影響 | 因應方式 |
| --- | --- | --- |
| 分攤規則未先定案 | 資料模型與計算反覆修改 | 在階段 0 確認規則並記錄決策 |
| 金額使用浮點數 | 出現累積誤差與帳目不平 | 全程使用最小貨幣單位整數並測試差額 |
| 付款與應收直接綁死 | 無法支援部分或合併付款 | 使用付款分配關聯表 |
| 停用或刪除破壞歷史 | 統計與稽核無法核對 | 採停用／封存，限制有歷史資料的刪除 |
| RLS 漏設或只靠前端 | 未授權資料存取 | Migration 建立 RLS，加入資料庫層測試 |
| GitHub Pages 路由錯誤 | 重新整理後出現 404 | 提前決定路由策略並做部署驗證 |
| 高權限金鑰外洩 | 資料庫遭完整存取 | 前端僅用公開金鑰，掃描 Repository 與產物 |
