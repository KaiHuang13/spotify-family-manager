# 需求規格

## 1. 專案目標

建立「Spotify Premium Family 訂閱分攤管理系統」，讓單一管理者能集中掌握訂閱成員、分攤費用、帳期、付款狀態、付款紀錄及歷史異動，降低人工試算與追蹤遺漏。

## 2. 第一階段使用者

### 管理者

- 第一階段唯一的系統使用者與資料擁有者。
- 需先透過 Supabase Auth 登入，才能存取管理資料。
- 可新增、查看、修改及停用管理範圍內的資料。
- 由管理者自行確認收款結果並手動登記付款。

第一階段不提供訂閱成員登入或自行回報付款的入口。

## 3. 第一階段功能需求

### 3.1 訂閱狀態儀表板

- 顯示目前方案與訂閱狀態。
- 顯示有效成員數與可用名額。
- 顯示目前帳期應收、已收、未收金額。
- 顯示待付款或逾期項目摘要。
- 顯示近期帳期與付款動態。

### 3.2 訂閱成員管理

- 建立與維護成員基本資料。
- 設定成員加入日、離開日及啟用狀態。
- 設定成員採月繳或年繳。
- 查看個別成員目前費用、付款狀態與歷史紀錄。
- 保留已離開成員的歷史資料，不直接刪除相關帳務紀錄。

### 3.3 月繳與年繳設定

- 可為成員指定月繳或年繳週期。
- 可設定方案總費用與費用生效時間。
- 可記錄成員的分攤金額或分攤依據。
- 費用變更後應保留舊帳期的原始金額，不回溯覆寫歷史帳務。

### 3.4 帳期與費用分攤

- 依成員付款週期產生或維護應收帳期。
- 每個帳期應記錄起訖日、應付金額、到期日及狀態。
- 可查看特定帳期的應收明細與加總。
- 應能處理成員於帳期中加入或離開的情境；實際計費方式待確認。

### 3.5 付款狀態及付款紀錄

- 管理者可手動登記付款日期、金額、方式與備註。
- 可區分未付款、部分付款、已付款及逾期等狀態。
- 同一筆應收可保留一筆或多筆付款紀錄；是否允許多筆付款待確認。
- 更正付款資料時須保留足以追查異動的歷史資訊。

### 3.6 統計資料

- 統計指定期間的應收、實收與未收金額。
- 統計月繳與年繳成員數。
- 統計目前有效、已離開及欠款成員數。
- 提供按月份或年度查看的彙總資訊。

### 3.7 歷史紀錄

- 查看成員加入、離開與設定變更紀錄。
- 查看費用、帳期及付款歷史。
- 歷史資料應呈現事件發生時間及相關內容。
- 是否需要完整記錄每次修改者、修改前後值，待確認。

## 4. 非功能需求

### 安全與資料隔離

- 未登入者不得讀取或修改管理資料。
- 所有資料表須啟用 Supabase Row Level Security。
- 存取規則須以已登入管理者的身分為基礎，不可只依賴前端隱藏功能。
- 前端只能使用可公開的 Supabase 設定；不得放入 service role key 或其他高權限密鑰。

### 可維護性與擴充性

- 使用 React、TypeScript 與 Vite。
- 第一階段介面與規則聚焦 Spotify Premium Family。
- 資料與模組邊界應避免把 Spotify 專屬名稱綁死於所有核心結構，以利未來擴充其他訂閱服務。
- 金額運算與顯示需採一致的幣別及精度規則。

### 部署與相容性

- 前端部署至 GitHub Pages，須支援其靜態網站與子路徑環境。
- 前端路由策略須避免重新整理頁面時出現 404。
- 主要使用情境以桌面瀏覽器為主，並應具備基本行動裝置可用性；支援範圍待確認。

### 可追溯性

- 重要帳務資料需有建立與更新時間。
- 已形成歷史依據的帳期與付款資料，不應因目前設定改變而失去當時內容。

## 5. 第一階段不包含

- 建立 React 專案或任何功能實作（目前文件規劃階段）。
- 串接信用卡、轉帳、電子支付或其他金流服務。
- 自動確認入帳或銀行對帳。
- 訂閱成員登入、自助查詢或付款回報。
- 通知、催繳訊息或電子郵件自動寄送。
- 管理 Spotify 帳號、邀請連結或呼叫 Spotify API。
- 管理 Spotify 以外的訂閱服務。
- 多管理者、角色權限或組織協作。

## 6. 第一階段完成條件

- 管理者能安全登入及登出。
- 管理者能完成成員從加入、計費、付款到離開的完整管理流程。
- 月繳及年繳成員皆能形成可追蹤的帳期與應收資料。
- 儀表板、統計及歷史紀錄能由同一套帳務資料一致計算。
- 未登入或不符合資料擁有者身分的請求會被 RLS 拒絕。
- 應用程式可從 GitHub Pages 正常載入及操作。
- 本文件與業務規則中的第一階段必須項目均已驗收。
