# 決策紀錄

## 使用方式

本文件記錄已確認且會影響產品、架構或資料的決策。每項決策使用唯一編號；已採用的決策不得直接刪除或改寫歷史，若需變更，應新增一項決策並標示取代關係。

狀態可使用：`已採用`、`待確認`、`已取代`、`不採用`。

## 已採用決策

### DEC-001：第一階段為單一管理者系統

- 狀態：已採用
- 日期：2026-07-21
- 決策：第一階段只有一位管理者使用，不建立多角色或成員登入入口。
- 理由：先聚焦訂閱與帳務管理的核心需求。
- 影響：資料仍保留擁有者欄位並使用 RLS，避免未來擴充時需要重建授權模型。

### DEC-002：前端採 React 與 TypeScript

- 狀態：已採用
- 日期：2026-07-21
- 決策：前端使用 React；所有功能程式碼使用 TypeScript。
- 理由：符合既定技術方向並提供型別檢查能力。
- 影響：不得在未經決策審查下改用其他前端框架或新增 JavaScript 功能程式碼。

### DEC-003：前端部署至 GitHub Pages

- 狀態：已採用
- 日期：2026-07-21
- 決策：第一階段以前端靜態網站方式部署至 GitHub Pages。
- 理由：符合初期單一管理者與低維運需求。
- 影響：前端路由、資源 base path 與 Supabase Auth redirect URL 必須相容 Repository Pages 網址；需要伺服器秘密的邏輯不能放在 GitHub Pages。

### DEC-004：驗證與資料庫採 Supabase

- 狀態：已採用
- 日期：2026-07-21
- 決策：使用 Supabase Auth 與 Supabase PostgreSQL，所有業務資料表以 RLS 保護。
- 理由：在靜態前端部署限制下提供託管驗證與資料庫能力。
- 影響：前端僅使用公開用戶端金鑰；`service_role` key 不得進入前端、Repository 或 GitHub Pages。資料庫變更必須建立 Migration。

### DEC-005：第一階段只管理 Spotify Premium Family

- 狀態：已採用
- 日期：2026-07-21
- 決策：畫面與流程只需支援 Spotify Premium Family。
- 理由：縮小第一階段範圍。
- 影響：資料命名避免完全綁死 Spotify，以保留未來擴充其他服務的可能；第一階段不實作服務切換或多服務操作。

### DEC-006：第一階段不串接金流

- 狀態：已採用
- 日期：2026-07-21
- 決策：付款由管理者手動登記，不串接金流、銀行或電子支付。
- 理由：先驗證帳期、分攤與追蹤流程。
- 影響：付款紀錄代表管理者確認的事實；系統不宣稱款項已由第三方自動驗證。

### DEC-007：金額以最小貨幣單位整數處理

- 狀態：已採用
- 日期：2026-07-21
- 決策：所有金額以最小貨幣單位整數儲存、傳輸與計算，禁止使用浮點數。
- 理由：避免二進位浮點誤差導致帳目不平。
- 影響：資料庫欄位使用整數型別並搭配貨幣代碼；顯示層負責格式化，不改變原始整數。

### DEC-008：有歷史資料的成員採停用，不直接刪除

- 狀態：已採用
- 日期：2026-07-21
- 決策：只要成員已有訂閱期間、應收費用或付款關聯，就不得硬刪除，改以停用或封存保留。
- 理由：維持歷史統計、付款核對及資料關聯完整。
- 影響：成員需有狀態與停用時間；資料庫外鍵與刪除規則應防止歷史被連帶刪除。

### DEC-009：付款與應收費用採多對多分配

- 狀態：已採用
- 日期：2026-07-21
- 決策：付款與應收費用之間使用付款分配紀錄連結。
- 理由：需支援部分付款、一筆付款支付多期，以及一筆應收分次付清。
- 影響：付款狀態由付款分配加總與到期日推導；付款金額與已分配金額必須可核對。

### DEC-010：歷史金額保存快照

- 狀態：已採用
- 日期：2026-07-21
- 決策：帳期與應收費用保存產生當下的金額與計算依據，不因日後方案或價格更新而回溯改寫。
- 理由：確保歷史報表可重現與核對。
- 影響：價格變更以新的生效期間或新紀錄表示，不覆寫已產生費用。

### DEC-011：成員與繳費設定分階段建立

- 狀態：已採用
- 日期：2026-07-22
- 決策：新增成員時先建立成員主檔與訂閱期間，月繳／年繳、分攤金額、貨幣及帳務起算基準可在後續流程一次補齊。
- 理由：新增成員不應被尚未確認的繳費方式阻擋，也不得以零元月繳等假資料滿足資料庫必填欄位。
- 影響：`members.joined_on` 保存成員加入日期；`subscription_members.start_date` 保存計費起算日；`payment_frequency`、金額、貨幣與 `billing_anchor_date` 必須同時為空或同時設定。後續設定帳務資料時必須以單一受控操作完整填入，不得留下部分設定狀態。

### DEC-012：Yearly 表示一次預繳十二個月

- 狀態：已採用
- 日期：2026-07-22
- 決策：成員繳費週期支援 `monthly` 與 `yearly`。`yearly` 只代表成員一次預繳十二個月，`monthly_share_minor` 始終保存每月應分攤金額。
- 理由：將每月成本歸屬與實際收款頻率分開，歷史帳期仍能按月呈現一致的分攤金額。
- 影響：`cycle_amount_minor` 在 monthly 時等於單月金額，在 yearly 時等於每月金額乘以十二。本次只設定週期與金額，不建立付款、不分配預繳款，也不實作跨月抵扣。

### DEC-013：月度帳期以開始日資格建立單月應收

- 狀態：已採用
- 日期：2026-07-22
- 決策：帳期由管理者手動建立且固定跨一個日曆月。只為帳期開始日當時有效的成員訂閱建立應收，每筆金額使用該訂閱期間的 `monthly_share_minor` 快照。
- 理由：以明確日期切點產生可重現的應收，避免自動排程或期間中途異動造成不一致。
- 影響：帳期開始日後才加入的成員不列入該期；開始日當時有效、但期間中途退出的成員仍收取完整單月分攤。本階段不按日比例，也不因 `yearly` 自動套用預繳抵扣。重疊或重複帳期由資料庫拒絕。

### DEC-014：手動付款直接對應單一成員應收

- 狀態：已採用
- 日期：2026-07-22
- 決策：第一階段每次手動登記建立一筆獨立 `payments`，直接對應一筆 `member_charges`；同一應收可有多筆付款。
- 理由：支援分次付款，並保持實際收款事件與原始應付快照分離。
- 影響：付款不更新 `member_charges.amount_minor`。一筆付款跨多筆應收、Yearly 預繳分配與抵扣仍留待 `payment_allocations` 實作。

### DEC-015：計費期間在介面稱為月份費用

- 狀態：已採用
- 日期：2026-07-23
- 決策：需求與資料語意使用「計費期間」，管理介面使用「月份費用」；付款期限與計費結束日分別呈現。
- 理由：「帳期」不夠直觀，且容易混淆費用涵蓋範圍與付款期限。
- 影響：既有資料表與欄位名稱維持不變，以避免無必要的結構更名；介面、說明與後續文件採用新詞彙。

### DEC-016：月份費用使用清單與獨立明細頁

- 狀態：已採用
- 日期：2026-07-23
- 決策：月份費用以清單呈現摘要，每筆月份費用由「查看明細」進入獨立頁面，不使用持續增長的下拉選單。
- 理由：避免歷史月份增加後選單難以瀏覽，並讓摘要與單月核對各自聚焦。
- 影響：第一階段以應用程式內頁面狀態切換，待 GitHub Pages 路由策略定案後再決定是否映射為網址路由。

### DEC-017：區分 Spotify 方案、成員加入與計費起算日期

- 狀態：已採用
- 日期：2026-07-23
- 決策：`subscriptions.started_on` 為整體 Spotify 方案開始日，`members.joined_on` 為成員實際加入 Family 日期，`subscription_members.start_date` 為該成員計費起算日。
- 理由：三個日期描述不同事件，不得以「訂閱開始日期」混稱。
- 影響：方案開始日在方案設定頁管理；成員清單分別顯示加入日期與計費起算日。

### DEC-018：新成員從實際加入日開始計費

- 狀態：已採用
- 日期：2026-07-23
- 決策：新增成員時，計費起算日預設且固定為本次輸入的實際加入日期。
- 理由：目前不採次月起算或自動按日折算，先以真實加入日作為資格邊界。
- 影響：建立月份費用時仍依 DEC-013，只納入計費開始日當時已開始計費的成員；月中加入者不會自動產生當月應收，日後若需補收須另行設計人工調整流程。
