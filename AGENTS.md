# Repository 工作規範

本文件適用於本 Repository 內所有開發、文件與資料庫工作。若任務說明與本文件衝突，應先停止並向需求提出者確認，不得自行擴大範圍。

## 工作方式

1. 一次只處理一個明確、可驗證的任務；若需求包含多個目標，先拆分並確認本次範圍。
2. 修改前先檢查現有目錄、文件、設定、資料模型與相關實作，避免重複建立或破壞既有架構。
3. 僅修改完成任務所需的檔案，不得順手重構或加入未要求的功能。
4. 不得無故更換既定技術、套件或架構。若確有必要，須先在 `docs/DECISIONS.md` 記錄理由、替代方案、影響，並取得同意。
5. 遇到需求不明、資料可能遺失或安全風險時，先停止並提出問題，不得自行猜測高風險決策。

## 技術與程式碼規範

- 前端技術維持 React 與 TypeScript；所有程式碼均使用 TypeScript，不新增 JavaScript 功能程式碼。
- 部署目標維持 GitHub Pages，後端服務維持 Supabase Auth 與 PostgreSQL。
- 不得使用浮點數處理金額。金額必須以最小貨幣單位的整數儲存與運算，例如新臺幣 100 元儲存為 `100`；若貨幣具有小數，則以分等最小單位儲存。
- 前端僅可使用 Supabase 公開用戶端金鑰。不得將 Supabase `service_role` key、資料庫密碼或其他高權限密鑰放入前端程式碼、建置產物、Git Repository 或 GitHub Pages。
- 任何資料庫結構、索引、限制條件、函式、觸發器或 RLS Policy 變更，都必須建立可追蹤且可重現的 Migration，不得只在遠端控制台手動修改。
- 已有歷史資料的成員不得直接刪除；必須以停用或封存方式保留其訂閱、帳期、費用與付款歷史。
- 新增資料存取功能時，必須同時檢查授權與 RLS，不能只依賴前端畫面隱藏。
- 日期與帳期邊界必須明確；資料庫日期優先使用 ISO 8601 語意，避免依瀏覽器本地時間推算帳期。

## 文件與決策

- 產品範圍以 `docs/PRODUCT_REQUIREMENTS.md` 為準。
- 開發順序與交付條件以 `docs/DEVELOPMENT_PLAN.md` 為準。
- 已確認的技術與產品決策記錄於 `docs/DECISIONS.md`。
- 資料模型規劃以 `docs/DATA_MODEL_DRAFT.md` 為起點；正式實作仍須透過 Migration。
- 若實作使上述文件失效，必須在同一任務中更新相關文件。

## 完成前驗證

每次完成修改後，必須執行專案定義的下列檢查：

1. `lint`
2. `typecheck`
3. `test`

不得因修改看似簡單而略過。若專案尚未提供某項指令、環境無法執行或檢查失敗，必須明確回報原因與影響，不得宣稱驗證通過。

## 完成回報

每次回報至少包含：

- 本次完成的任務與行為變更摘要。
- 修改或新增的檔案清單。
- `lint`、`typecheck`、`test` 各自的執行結果；未執行時需說明原因。
- 尚未處理事項、已知限制、風險或需要後續確認的決策。
